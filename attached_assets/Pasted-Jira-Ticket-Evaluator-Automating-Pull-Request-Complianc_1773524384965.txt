1. OVERVIEW
Modern software teams rely on Jira to capture requirements, track work, and communicate intent. Yet verifying whether a Pull Request (PR) actually satisfies the requirements described in its linked Jira ticket is still a largely manual, error-prone task — one that slows code review and lets requirement drift slip into production.
This hackathon challenges participants to build an AI-powered Jira Ticket Evaluator: a system that takes a Jira ticket and a GitHub PR as inputs, then autonomously decides whether the code changes satisfy the stated requirements.
2. PROBLEM STATEMENT
Given:
▪ A Jira ticket describing a feature, bug fix, or improvement — including acceptance criteria, description, and any attached specifications.
▪ A GitHub Pull Request (PR) containing the corresponding code changes, commit messages, and PR description.
Build a system that:
▪ Parses and understands the requirements captured in the Jira ticket.
▪ Retrieves and analyses the diff, file changes, and metadata of the GitHub PR.
▪ Produces a structured verdict — Pass / Partial / Fail — with clear reasoning for each requirement.
▪ Optionally executes or generates custom validation tests to provide additional evidence for its verdict.
3. TECHNICAL REQUIREMENTS
3.1 Mandatory Technologies
All submissions must make use of the following:
▪ GitHub — The PR (and optionally the full repository) must be hosted on GitHub. Participants will interact with the GitHub API and/or GitHub MCP server to retrieve PR data.
▪ MCP (Model Context Protocol) — At least one MCP server must be integrated to provide tool-use capability to the agent (e.g., the GitHub MCP server, a custom Jira MCP server, or a file-system MCP server for code access).
▪ AI Agents — The core evaluation logic must be agent-driven. A single LLM call is not sufficient; the system must demonstrate multi-step reasoning, tool invocation, and structured output.
▪ AI APIs — Use any publicly available LLM API (OpenAI, Anthropic, Google Gemini, etc.) for the reasoning layer.
3.2 Optional Enhancements
▪ Custom test generation — the agent writes and executes tests (unit, integration, or static analysis) to validate specific acceptance criteria.
▪ Multi-agent architectures — separate agents for parsing, retrieval, test execution, and verdict synthesis.
▪ Support for additional ticket sources (Linear, Shortcut, GitHub Issues, etc.).
▪ A web UI or CLI dashboard for displaying results.
▪ Confidence scoring and traceability — mapping each verdict point back to specific lines of code and ticket requirements.

4.2 Expected Output
▪ Overall verdict: Pass / Partial / Fail
▪ Per-requirement breakdown: each requirement from the ticket evaluated individually.
▪ Evidence: specific file paths, line numbers, or code snippets supporting each verdict point.
▪ Test results (optional): output from any custom tests the system generated or executed.
▪ Confidence score (optional): a numeric measure of the agent's certainty.

6.3 Presentation Deck Requirements
The slide deck must include all of the following:

▪ System architecture diagram (your own, not a template).
▪ Screenshots or recordings of the system in action with real Jira tickets and PRs.

▪ Challenges encountered and how they were resolved.

Note: All images, screenshots, and diagrams must be original or attributed. Presentations that use placeholder images or stock slides will be penalised during judging.
7. RULES & CONSTRAINTS
▪ Teams must use at least one MCP server for tool-use. Calling the LLM API directly without MCP-mediated tool invocation does not satisfy this requirement.
▪ The system must handle a minimum of three distinct Jira ticket types: feature request, bug report, and refactor/cleanup.
▪ No hardcoded logic — the evaluation must be generalised. Systems that only work on the specific example tickets provided during judging will not score above partial marks.
▪ API keys must not be committed to the repository. Use environment variables and provide a .env.example file.
▪ All dependencies must be declared in a package manager manifest (package.json, requirements.txt, pyproject.toml, etc.).
▪ Teams may use any programming language, LLM provider, or MCP server they choose.
